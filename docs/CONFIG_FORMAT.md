# Config Format - Capture Flag

## Config Format

Decision: use an in-house SDK and an in-house versioned Config JSON from the start. Compatibility with official ConfigCat SDKs is outside the initial scope.

The MVP uses automatic publishing: changing a flag in the client updates the config revision for that `config + environment` pair.

Initial example:

```json
{
  "schemaVersion": 1,
  "projectKey": "my-project",
  "configKey": "frontend-web",
  "environment": "production",
  "revision": 42,
  "generatedAt": "2026-05-12T00:00:00.000Z",
  "segments": {
    "beta-users": {
      "conditions": [
        {
          "attribute": "email",
          "operator": "endsWith",
          "value": "@example.com"
        }
      ]
    }
  },
  "flags": {
    "accountEnabled": {
      "type": "boolean",
      "defaultValue": true,
      "rules": [],
      "percentageAttribute": "identifier",
      "percentageOptions": []
    },
    "newCheckout": {
      "type": "boolean",
      "defaultValue": false,
      "rules": [
        {
          "conditions": [
            { "segment": "beta-users" },
            { "prerequisiteFlag": "accountEnabled", "operator": "equals", "value": true },
            { "attribute": "custom.tags", "operator": "arrayContains", "value": "beta" },
            { "attribute": "custom.releaseDate", "operator": "dateAfter", "value": "2026-05-12" },
            { "attribute": "custom.appVersion", "operator": "semverGreaterThan", "value": "1.2.3" }
          ],
          "serve": true
        }
      ],
      "percentageAttribute": "identifier",
      "percentageOptions": []
    },
    "themeConfig": {
      "type": "json_object",
      "defaultValue": {
        "theme": "dark",
        "enabledModules": ["checkout", "billing"]
      },
      "rules": [],
      "percentageAttribute": "identifier",
      "percentageOptions": []
    }
  }
}
```

Supported types in `flags[*].type`: `boolean`, `string`, `integer`, `double`, `json_object`, and `json_array`. `json_object` requires a JSON object as the root; `json_array` requires a JSON array as the root. JSON values are preserved in `defaultValue`, `rules[*].serve`, and `percentageOptions[*].value` without serializing them as strings.

The `ETag` must be exposed as an HTTP header and derived from the revision or content. The public endpoint must accept `If-None-Match` and respond with `304 Not Modified` when the config has not changed.

The versioned public endpoint for SDKs is `GET /public-api/v1/sdk/:sdkKey/config`. It does not use a session; the raw SDK key is treated as a credential, hashed for lookup, and never stored in plaintext.

Percentage rollout must be deterministic across SDKs. The bucket is calculated with unsigned 32-bit FNV-1a over the string `${flagKey}:${attributeValue}` encoded as UTF-16/JavaScript units, using offset basis `2166136261` and prime `16777619`. The unsigned result (`hash >>> 0`) must be reduced modulo `10000`, producing an integer bucket `0..9999`. Percentages use basis points: `1% = 100` units, `0.01% = 1` unit, and precision above two decimal places is invalid. When `percentageOptions` is not empty, the sum must be exactly `100%`. The first option whose cumulative limit is greater than the bucket wins.

`segments` is optional for old caches, but configs generated by the current API must send it as an object. Each segment contains `conditions`, using the same attribute-based targeting operators. Rules reference segments with a condition `{ "segment": "segment-key" }`. Segments cannot reference other segments in Phase 6.

Phase 7 advanced targeting expands conditions without changing `schemaVersion`:

| Condition | Shape | Note |
|---|---|---|
| Segment reference | `{ "segment": "beta-users" }` | Only in rules; segments cannot nest segments |
| Prerequisite flag | `{ "prerequisiteFlag": "accountEnabled", "operator": "equals", "value": true }` | Only `equals` and `notEquals`; value must match the primitive type of the referenced flag; JSON flags are not valid prerequisites |
| Array contains | `{ "attribute": "custom.tags", "operator": "arrayContains", "value": "beta" }` | The Evaluation Context attribute must be an array |
| Date comparison | `{ "attribute": "custom.releaseDate", "operator": "dateAfter", "value": "2026-05-12" }` | `dateBefore` and `dateAfter`; accepts a numeric timestamp or ISO `YYYY-MM-DD`/date-time string with timezone |
| SemVer comparison | `{ "attribute": "custom.appVersion", "operator": "semverGreaterThan", "value": "1.2.3" }` | Operators: `semverEquals`, `semverGreaterThan`, `semverGreaterThanOrEquals`, `semverLessThan`, `semverLessThanOrEquals`; requires a SemVer 2.0.0 `MAJOR.MINOR.PATCH` string; build metadata is ignored; prerelease follows SemVer precedence |

Prerequisite flags are evaluated locally by the SDK using the same Config JSON. Cycles are rejected by the API and treated as non-match by the evaluator to protect SDKs from invalid configs. `json_object` and `json_array` can be served values, but they do not participate in prerequisite comparison semantics.

This path reduces complexity and allows the SDK to evolve without depending on another product.
